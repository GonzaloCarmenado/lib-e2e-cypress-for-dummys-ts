# 002 — Test Notes Field

> **Status:** In Progress
> **Date:** 2026-05-29
> **Author:** Gonzalo

---

## Context and motivation

When a test is saved, the only identifying information is the short `it()` name (e.g. "Login correcto"). There is no place to capture longer documentation: what the test validates, preconditions, or caveats. Reviewers reading a Cypress file also have no inline context unless they manually add comments.

This spec adds a free-form **notes** textarea to the save flow so users can document a test at creation time. Notes are stored in IndexedDB alongside the test record and, when the test is inserted into a Cypress file, are emitted as a `/** … */` block comment directly above the `it()` block.

---

## Use cases

1. **UC-01 — Add notes when saving a recording**
   As a tester, I want to write a multi-line description while saving a recorded test so that future readers understand what the test validates without opening the source.

2. **UC-02 — See notes in the test list**
   As a tester, I want to see the notes for a saved test when I expand it in the test-editor panel so that I can quickly recall what the test covers.

3. **UC-03 — Notes as block comment in Cypress file**
   As a tester, I want the notes to appear as a `/** … */` block comment above the `it()` block when a test is inserted into a `.cy.ts` file so that the intent is self-documented in code.

## Acceptance criteria

- [ ] AC-01: `save-test` renders a `<textarea id="notes-input">` below the test-name input.
- [ ] AC-02: Notes value is included in the `savetest` and `saveandexport` event `detail.notes`.
- [ ] AC-03: `PersistenceService.insertTest` accepts an optional `notes` parameter and persists it on the test record.
- [ ] AC-04: `TestRecord` and `TestWithDetails` include `notes?: string`.
- [ ] AC-05: `getAllTests()` returns `notes` for every test.
- [ ] AC-06: `getTestById()` returns `notes` in the `TestDetail` object.
- [ ] AC-07: The test-editor expanded row shows notes text (plain, not code) when `notes` is non-empty.
- [ ] AC-08: `AdvancedTestTransformationService.buildBlockComment(notes)` returns a correctly formatted `/** … */` string.
- [ ] AC-09: When `notes` is non-empty, `saveCommandsToFile` inserts the block comment immediately above the `it()` block in the file.
- [ ] AC-10: All 5 i18n files contain the new keys (`SAVE_TEST.NOTES_LABEL`, `SAVE_TEST.NOTES_PLACEHOLDER`).
- [ ] AC-11: No DB version bump is required — `notes` is a new optional property on existing records; old records silently have `notes: undefined`.

## Public API changes

```typescript
// PersistenceService
insertTest(
  name: string,
  commands?: string[],
  interceptors?: string[],
  tags?: string[],
  notes?: string,          // NEW
): Promise<number>;

// TestRecord (private interface)
interface TestRecord {
  id: number; name: string; createdAt: number;
  tags?: string[];
  notes?: string;          // NEW
}

// TestWithDetails (public)
export interface TestWithDetails extends TestRecord {
  commands: string[];
  interceptors: string[];
  // notes inherited from TestRecord
}

// AdvancedTestTransformationService
buildBlockComment(notes: string): string;  // NEW — returns /** … */

// save-test CustomEvent detail
{ description: string | null; tags: string[]; notes: string }  // notes added
```

---

## Out of scope

- Editing notes after a test has been saved (future feature).
- Searching or filtering tests by notes content.
- Showing notes in the `describe()` block generated by multi-select.
- Showing notes in the test-previewer (live recording panel) — only in the saved test editor.

---

## Implementation notes

- No DB version bump: IndexedDB stores records as plain objects; adding `notes` to new records is backwards-compatible. Old records simply return `undefined` for `notes`.
- Block comment format:
  ```
  /**
   * First line of notes
   * Second line
   */
  ```
  Each line of the notes textarea is prefixed with ` * `. An empty notes field produces no comment.
- The block comment is inserted by `insertItBlock` (or a wrapper) so it lands in the file immediately before the `it()` block — not inside `describe` or `beforeEach`.
